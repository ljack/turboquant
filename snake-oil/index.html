0 The Problem: Memory Is the Bottleneck

LLMs are smart, but they're memory hogs. During inference, the KV cache can eat more RAM than the model weights themselves.

Why should you care?

When an LLM generates text, it stores keys and values for every token it has seen so far. A 70B parameter model serving a 128K context window can require over 40GB of memory just for the KV cache — and that's per request. At 1M tokens, you're looking at 300+ GB just for KV — impossible without compression.

This means: fewer concurrent users, bigger GPU bills, and hard limits on context length.

32K

This is why TurboQuant matters. It's not just an academic exercise — shrinking the KV cache by 6x means you can serve 6x more users on the same hardware, or fit contexts that were previously impossible. A 70B model at 1M context needs ~1.3 TB of KV cache in FP16 — that's a full server rack. With TurboQuant, it's ~220 GB — a single node. And it does this with zero retraining.

1 KV Cache 101

Before we compress it, let's understand what the KV cache actually is.

Attention in 30 seconds

Transformers use attention to decide which earlier tokens matter for predicting the next one. For each token, the model computes three vectors:

  • Query (Q) — "What am I looking for?"
  • Key (K) — "What do I contain?" (stored in cache)
  • Value (V) — "What information do I carry?" (stored in cache)

Attention scores = softmax(Q · KT / √d), then weighted sum of V.

Keys (K)

Each token's "label." When generating a new token, its Query is compared against all stored Keys to find relevant context. Inner product accuracy is critical here.

Values (V)

The actual content retrieved via attention. Once the model decides which tokens are relevant (via K), it pulls the corresponding Values to form its output.

8

The key insight for compression: We don't need the exact K and V vectors — we just need the inner products (Q·K) to be preserved accurately. TurboQuant exploits this: it's designed to minimize inner product error, not element-wise error. That's a fundamentally different (and easier) target.

2 Random Rotation (PolarQuant Step 1)

The first move is deceptively simple: multiply every vector by a random orthogonal matrix. This is the key that unlocks everything else.

What does rotation do?

Real KV vectors are spiky — some coordinates are large, others near zero. This makes quantization hard because you'd need different precision for each coordinate.

A random rotation spreads the energy uniformly across all coordinates. Think of it like shaking a jar of sand until it's level — the total amount stays the same, but it's evenly distributed.

Crucially, rotation preserves all inner products: Rx · Ry = x · y. So attention scores are unchanged.

3 The Beta Distribution Miracle

After rotation, something remarkable happens: every coordinate follows a known, predictable distribution.

Why Beta?

When you randomly rotate a unit vector in d dimensions, each coordinate follows a Beta(1/2, (d-1)/2) distribution (after rescaling). In high dimensions (d > 64, typical for LLM heads), this concentrates tightly around zero — almost Gaussian.

The magic: since we know the distribution ahead of time, we can precompute the optimal quantization grid before ever seeing any data.

64 3

This is where the information theory clicks. Most quantization methods treat coordinates as having unknown distributions and waste bits being conservative. TurboQuant knows the distribution (because the rotation made it predictable), so it can place quantization levels exactly where they minimize distortion. It's like the difference between a one-size-fits-all shoe and a custom mold — except the "custom mold" is computed from pure math, not data.

4 Lloyd-Max: Optimal Bucket Placement

Now that we know the distribution, we compute the mathematically optimal quantization levels.

What is Lloyd-Max?

Given a known probability distribution and a bit budget (say 3 bits = 8 levels), the Lloyd-Max algorithm finds the 8 reconstruction points and 7 decision boundaries that minimize mean squared error (MSE).

It's an iterative algorithm: place boundaries at midpoints between reconstruction levels, then move reconstruction levels to the centroid of their region. Repeat until converged.

Since the Beta distribution is known, this is computed once, offline, for each (d, bits) pair. Zero runtime cost.

Pre-computed = free at inference. Most quantization schemes run calibration passes that take minutes to hours. TurboQuant's quantizer is a lookup table baked at compile time. The only runtime operation is: rotate, then round to nearest level. That's why it's actually faster than running with full precision — less data to move through the memory bus.

5 QJL: 1-Bit Error Correction

Lloyd-Max minimizes MSE, but MSE-optimal quantizers introduce bias in inner product estimation. QJL fixes this with just 1 extra bit per dimension.

The Bias Problem

When you quantize a vector x to x̂, the error e = x - x̂ is correlated with x itself (values near boundaries get rounded systematically). This means E[x̂ · ŷ] ≠ x · y — the inner products are biased.

For attention, biased inner products mean the model might attend to the wrong tokens. Bad.

The QJL Fix

Quantized Johnson-Lindenstrauss (QJL) compresses the residual error to just 1 bit per dimension: store only the sign of each coordinate of a random projection of the error.

This gives an unbiased estimator of the error's inner product. Combined with the main quantizer: x · y ≈ x̂ · ŷ + correction.

Total cost: just 1 extra bit. Total memory overhead: negligible.

This is the theoretical crown jewel. The QJL estimator is provably unbiased, meaning errors cancel out in expectation. And it only costs 1 bit! Most error-correction schemes in quantization literature either cost too many bits to be worthwhile or require complex per-layer calibration. QJL is elegant: random projection → take sign → done. It's the mathematical equivalent of a checksum, but for inner products.

What about decompression?

At inference time, to compute Q · K:

  1. Rotate Q by the same R: Q' = RQ
  2. Compute Q' · K̂ using the quantized keys (integer arithmetic!)
  3. Add QJL correction term

Because quantized values are small integers, the dot products are much faster than float operations on modern GPUs with INT4/INT8 tensor cores.

The total package is unreasonably effective. Each piece — rotation, Beta quantization, QJL — is clean and well-understood on its own. But the combination achieves near-optimal compression (within 2.7x of the information-theoretic limit) with zero training, zero calibration, and faster inference. The rotation makes the distribution predictable, the predictable distribution makes quantization optimal, and QJL cleans up the only remaining artifact. No wasted bits anywhere.
